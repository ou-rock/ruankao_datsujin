# RAG 记忆与进步控制方案

## Grilling 结论

1. RAG 不做新的事实源。SQLite 仍保存三源材料、记忆卡、复习日志和练习记录。
2. v0 只有本地检索和进步控制，偏壳子；v1 必须具备切块、索引、FTS/BM25、混合排序、引用和排序解释。
3. RAG 不是搜索框，而是控制层：每次输出召回证据和进步闸门。
4. 进步闸门优先级高于资料解释。反复低分、到期复习、题型缺口和低分练习先处理。
5. NotebookLM、Cheko、文章和真题仍进入 Uns；RAG 只召回和组织，不替代判断。
6. 每次 RAG 使用后，必须产生下一张卡、一次练习或一条 Mein/Du/Uns 沉淀。

## 文档来源

- `raw_records`：Mein / Du / Uns 的原始材料和提炼状态。
- `memory_cards`：可检索记忆对象，带题型、强度、到期日期和卡片类型。
- `review_logs`：主动回忆历史，用于判断反复低分、不稳定和到期。
- `practice_sessions`：选择、案例、论文的实战记录、得分和错因。

## 排序原则

召回分数由多部分组成：

- 切块：把三源材料、记忆卡和练习记录拆成可引用的 `chunk_ref`。
- 索引：用本地 SQLite FTS5 建临时全文索引。
- BM25：用 FTS5 `bm25()` 得到 lexical retrieval 信号。
- 词项命中：查询词、中文二字词、题型匹配、标题和正文命中。
- 进步权重：反复低分、最近低分、到期复习、低分练习、未升格材料优先。
- 混合重排：把 BM25、词项命中、短语命中、题型、状态和进步权重合成为最终排序。

这意味着 RAG 不会只找“最像的问题”，还会把最阻碍进步的材料推到前面。

## 输出契约

RAG 简报必须包含：

- `进步闸门`：今天必须先处理的风险。
- `召回证据`：来自三源、记忆卡或练习记录的依据。
- `chunk_ref`：可追溯到原始文档和切块位置。
- `score_breakdown`：展示 progress、token、fts_bm25、phrase、front、status 的分数贡献。
- `建议动作`：下一步学习、复习或沉淀动作。
- `回答契约`：回答时必须如何引用证据并沉淀结果。

## 使用入口

```sh
python3 -m ruankao_agent.cli rag-query \
  --root /Users/pedan/Documents/ruankao/ruankao-agent \
  --query "质量属性场景为什么总丢分？" \
  --front case
```
